Project Information

Diffusion LLMs for
Automatic Code Refactoring

Project Overview

The introduce of LLMs has significantly improved the efficiency of software development. However, they are also accumulating significant technical debt over time. Code refactoring, the process of improving code structure without changing external behavior, is a critical but time-consuming engineering activity. Recent advances in Discrete Diffusion Language Models (dLLMs) open a new avenue: instead of autoregressively predicting tokens left-to-right, dLLMs iteratively refine entire code spans bidirectionally, making them uniquely well-suited to structured code transformation tasks.

This project will be formed up with two student groups, 5 student each. Every group will be assigned one topic and each student will have their own research question. Overall contribution and personal work will both be eavluated during the grading process.

Models Under Study

Each student will run experiments comparing at least one diffusion model against at least one autoregressive baseline relevant to their research question.

Diffusion Models
DiffuCoder Discrete dLLM fine-tuned on code corpora; primary focus of this project
DreamCoder Diffusion-based model combining program synthesis with neural guidance
Autoregressive Baselines
Qwen-Coder Alibaba's state-of-the-art code LLM with strong reasoning capability
CodeLlama Meta's open code LLM supporting fill-in-the-middle generation
CodeGemma Google DeepMind's code-specialized Gemma model

Student Groups & Research Questions

The cohort is divided into two groups, each tackling a distinct angle of the dLLM-for-refactoring hypothesis. All RQs in each group are equally important deliverables.

Group A

Prompt Sensitivity — Robustness of dLLMs vs. AR Models

Project Pitch: Minor prompt variations often cause severe output fluctuations in AR models during code refactoring tasks. Inspired by the Supervised Fine-Tuning (SFT) paradigm of Large Language Diffusion Models (e.g., LLaDA, as shown in Fig 2b), where the entire prompt remains fully unmasked and visible while only the response is masked and iteratively refined, this project empirically tests whether this unique global conditioning mechanism makes dLLMs significantly more robust to prompt perturbations than strictly left-to-right AR models.

How accurately do dLLMs and AR models map localized prompt instructions to specific code modifications, and how well do they isolate changes when partial prompts are perturbed?
Goal: Evaluate the structural alignment capabilities of both architectures. Since dLLMs process the unmasked prompt bidirectionally without causal masks, this project aims to test whether they can execute independent refactoring constraints (e.g., "rename X to Y" and "extract method Z") more precisely in their corresponding output segments than AR models, and whether perturbing one specific constraint causes cascading failures in unmodified regions.
To what extent does the spatial ordering of multi-intent prompts affect the localized structural mapping capabilities of dLLMs compared to unidirectional AR models?
Goal: Validate the theoretical hypothesis regarding recency bias. Since AR models generate left-to-right, they often overweight instructions placed at the end of a prompt. This RQ targets whether dLLMs, utilizing unmasked global prompt conditioning, yield order-invariant local code modifications regardless of how independent instructions (e.g., A then B vs. B then A) are shuffled in the prompt.
How resilient is the localized prompt-to-code mapping in dLLMs and AR models when semantic noise or irrelevant constraints are injected into the prompt?
Goal: Test the limits of attention dilution. Since a dLLM's mask predictor relies entirely on the fully unmasked prompt to denoise the masked response simultaneously, assess if injecting localized noise (e.g., irrelevant comments or conflicting pseudo-constraints) disrupts its global attention, causing it to misapply or forget valid localized refactoring tasks, compared to AR models filtering noise left-to-right.
Does the structural isolation of local refactoring constraints in dLLMs and AR models translate to global semantic correctness and compilation success?
Goal: Connect structural robustness to functional validity in SE. A model might successfully isolate a local change textually (e.g., renaming a variable only inside one loop, as tested in RQ1), but fail to resolve global semantic dependencies (e.g., failing to update the method signature, causing a compilation error). Evaluate if the bidirectional diffusion objective maintains better Abstract Syntax Tree (AST) integrity and compilation pass rates under local perturbations than AR baselines.
Group B

Tool for Refactoring Mining — The "Survival Time" of Technical Debt

Project Pitch: Through repository mining, we aim to uncover the underlying reasons and triggers that prompt human developers to perform code refactoring. Inspired by recent findings that bugs resembling AI-generated code survive significantly longer in repositories (taking more commits to resolve), this project investigates the "latency" or "survival time" of technical debt. The ultimate goal is to trace back using version history (e.g., git blame) to quantify precisely how much earlier a refactoring task could have been performed, and whether code originating from "other developers" (a proxy for AI assistants) disproportionately delays necessary structural improvements.

What are the primary triggers and evolutionary patterns in codebases that finally force developers to initiate a refactoring process?
Goal: Mine commit histories to identify the specific codebase conditions (e.g., code smell thresholds, complexity jumps, or file size limits) that historically break the camel's back and lead to manual refactoring.
How long is the "latency period" of technical debt, measured in the number of commits, from introduction to refactoring?
Goal: Borrowing the methodology of tracking "commits to fix", trace the history of refactored files to establish the exact timeline of technical debt accumulation. Quantify the delay (in days and commit counts) between when a code smell was first introduced and when it was finally refactored.
Does the authorship of suboptimal code (Self vs. Other/AI) significantly extend its survival time before refactoring?
Goal: Test the hypothesis derived from AI-assisted programming studies: since AI acts as an "other developer" whose code is often poorly understood by the active user, investigate whether code snippets introduced by different authors (or identified as AI-like patterns) survive significantly more commits before being refactored compared to self-authored code.
Can we predict the "optimal intervention point" for automatic refactoring tools based on historical latency data?
Goal: Use the data generated in RQ1 and RQ2 to define a theoretical "intervention window." Determine how much earlier an AI-driven refactoring tool should suggest changes to prevent the exponential accumulation of technical debt, minimizing the long commit delays observed in human-driven workflows.

General Methodology

1. Benchmark Construction

Identify and prepare a dataset of before/after refactoring pairs from real-world developer commits. Align it to your specific research question (structural or quality refactoring).

2. Context Extraction & Prompting

Extract sufficient code context. For dLLMs, use masking strategies targeting the region to be refactored. For AR baselines, design appropriate instruction or FIM prompts.

3. Model Inference

Run your selected dLLM and AR models on the benchmark. Log all outputs, runtimes, and any failure cases (OOM, invalid output, etc.).

4. Evaluation

Evaluate using metric(s) appropriate to your RQ: CodeBLEU, exact match, test-suite pass rate, static analysis scores, or LLM-as-judge. Discuss quantitative and qualitative results.
